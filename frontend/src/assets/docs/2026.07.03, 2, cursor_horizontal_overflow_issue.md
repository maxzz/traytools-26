The horizontal overflow on **Windows Tree** comes from a broken **flex `min-width` chain**, not from the page wrapper itself.

## Why only this page?

Trace Manager and Tools Menu Editor don’t hit the same constraints:

- **Trace Manager** uses the same resizable split, but its content (lists, checkboxes) stays relatively narrow.
- **Tools Menu Editor** has no horizontal split — just a vertical `ScrollArea`.
- **Windows Tree** has two resizable panels with **long monospace tree labels**, a filter row with `min-w-40`, and a **4-tab `TabsList`** (`w-fit` + `whitespace-nowrap`). Any of those can force a flex child wider than the viewport.

In flex layouts, children default to `min-width: auto`, so they won’t shrink below their content width. Wide content pushes the panel → panel group → page outward, causing a page-level horizontal scrollbar.

Your page wrapper already had `min-w-0 overflow-hidden`, but the **panels and their inner columns** did not, so content could still escape.

## Page-only fix

All changes stay under `1-tab-windows-tree/`:

1. **`0-all-windows-tree.tsx`** — `min-w-0` on `ResizablePanelGroup` and both `ResizablePanel`s so panels can shrink.
2. **`1-0-windows-tree.tsx`** — `min-w-0 overflow-hidden` on the tree column; filter input `min-w-40` → `min-w-0`.
3. **`2-window-props.tsx`** — same on the props column; `TabsList` gets `max-w-full overflow-x-auto` so tabs scroll instead of widening the page.
4. **`1-1-tree-node.tsx`** — `min-w-0` on `TreeNodeTrigger` so `TreeLabel`’s `truncate` actually works.

The pattern is: **`min-w-0` at every flex/resizable level**, plus **`overflow-hidden`** on panel content roots, so overflow stays inside each panel’s `ScrollArea` instead of expanding the whole app.
