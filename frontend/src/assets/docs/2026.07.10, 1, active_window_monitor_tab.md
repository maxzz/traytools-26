## What it does

The legacy `liswatch_t::on_timer()` polled the local input state every 500 ms and displayed the **Foreground**, **Active**, **Focus**, and **Capture** windows plus the attached thread. Because `GetActiveWindow`/`GetFocus`/`GetCapture` return *per-thread* queue state, the original code used `AttachThreadInput` to attach to the foreground window's thread before reading them. I reproduced that exact dance in Go.

## Changes

**Backend (Go — `backend/windowtree/`)**
- `types.go`: added `MonitorWindow` and `ActiveWindowsInfo` types (with `valid`/`noWindow` flags matching the old `calcwindowtext()` "no window"/"invalid window" cases).
- `enum_windows.go`: added `GetForegroundWindow`, `GetActiveWindow`, `GetFocus`, `GetCapture`, `AttachThreadInput` procs and `platformGetActiveWindows()` — the AttachThreadInput logic ported from `liswatch_t::on_timer()`. It reuses the existing class-name/title/thread helpers.
- `platform_other.go`: non-Windows fallback so the tab stays exercisable in browser/Vite dev.
- `manager.go`: registered the `getActiveWindows` command on the existing `windowtree` bus group.

## Design notes
- I kept it as a **polling** feature (frontend `setInterval` → bus call) rather than backend push events, which mirrors the original timer model and needed no new event infrastructure.
- I added it to the existing `windowtree` group since it reuses the same Win32 helpers, rather than creating a whole new package/group.
