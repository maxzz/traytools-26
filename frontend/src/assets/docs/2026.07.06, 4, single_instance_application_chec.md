Single-instance enforcement is in place using Wails’ built-in `SingleInstanceLock`, plus an early Windows check so a duplicate launch activates the running app instead of triggering UAC elevation again.

### How it works

1. **Early exit (Windows)** — `EnsureSingleInstanceOrExit()` runs before elevation. If another instance is already running, it sends a message to that instance and exits immediately.
2. **Wails single-instance lock** — `SingleInstanceLock` ensures only one process owns the app mutex on Windows, Linux, and macOS.
3. **Window activation** — When a second launch is detected, `OnSecondInstanceLaunch` calls the existing `showWindow()` helper, which unminimizes, shows, and focuses the main window.

The single-instance check had several problems that together allowed duplicate launches. Here’s what was wrong and what changed.

### Root causes

1. **Dev-mode bypass** — Single-instance was disabled for `wails dev` binaries (same as elevation), so launching twice in dev always started two processes.
2. **Unreliable Wails-only lock** — Wails’ hidden message-only window isn’t reliably found with `FindWindow`, and when that failed the second process kept starting.
3. **Blocking IPC deadlock** — The second instance used synchronous `SendMessage`, which could block until the first instance handled the message; if the UI thread was busy, the second process never reached `os.Exit(0)`.

### Fix

Replaced the approach with a dedicated Windows single-instance implementation:

- **Named mutex** held for the process lifetime (with a permissive security descriptor so elevated/non-elevated launches share the same lock)
- **Early check** in `EnsureSingleInstanceOrExit()` before elevation relaunch
- **Second instance** always exits after notifying the running instance
- **Activation** via:
  - `PostMessage` with a registered window message (non-blocking, safe across processes)
  - Direct `ShowWindow` / `SetForegroundWindow` on the main `"traytools"` window as fallback
- **Removed the dev-mode bypass** — single-instance now applies in both dev and release builds

Verified with rapid triple-launch testing: only **1 process** remains running.
