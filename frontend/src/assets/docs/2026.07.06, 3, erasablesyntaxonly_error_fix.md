The build fails because **Wails-generated bindings use TypeScript `namespace` syntax**, which conflicts with `erasableSyntaxOnly: true` in your tsconfig.

## Root cause

`frontend/wailsjs/go/models.ts` is auto-generated by Wails and contains:

```typescript
export namespace options {
    export class SingleInstanceLock { ... }
}
```

With `erasableSyntaxOnly: true` (TypeScript 5.8+), namespaces and enums are disallowed because they emit runtime JavaScript. That triggers **TS1294**.

Even though `wailsjs` is listed in `exclude` in `tsconfig.app.json`, TypeScript still type-checks it because your app imports from it (e.g. `wailsjs/go/backend/App.d.ts` imports `../models`).

## Fix options

### Option 1: Remove `erasableSyntaxOnly` (recommended for Wails)

Remove this line from `frontend/tsconfig.app.json`:

```json
"erasableSyntaxOnly": true,
```

This is the most practical fix today. Wails does not yet generate fully erasable TypeScript ([wailsapp/wails#4531](https://github.com/wailsapp/wails/issues/4531)).

### Option 2: Keep `erasableSyntaxOnly` and patch the generated file

After each `wails generate` / `wails dev` run, change `models.ts` from a runtime namespace to a `declare` namespace:

```typescript
export class SingleInstanceLock {
    UniqueId: string;

    static createFrom(source: any = {}) {
        return new SingleInstanceLock(source);
    }

    constructor(source: any = {}) {
        if ('string' === typeof source) source = JSON.parse(source);
        this.UniqueId = source["UniqueId"];
    }
}

export declare namespace options {
    export { SingleInstanceLock };
}
```

`declare namespace` is type-only and passes `erasableSyntaxOnly`. The downside: Wails will overwrite this file on the next binding regeneration, so you’d want a small post-generate script.

### Option 3: `wails.json` interfaces mode (partial help)

You can add to `wails.json`:

```json
"bindings": {
  "ts_generation": {
    "outputType": "interfaces"
  }
}
```

That switches classes to interfaces, but Wails still wraps them in `export namespace`, so it does **not** fully fix TS1294 on its own.

---

**Recommendation:** Use **Option 1** unless you specifically need `erasableSyntaxOnly` for Node’s native TypeScript runner. Wails-generated bindings are not compatible with that flag yet.

If you want, I can apply Option 1 (remove the flag) or set up a post-generate patch script for Option 2.
