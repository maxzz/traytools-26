# Windows shortcut launch issue

**User**

When launching the application via a Windows shortcut, a problem arises where the window appears in inconsistent positions—sometimes even starting minimized. 
However, if the application is launched directly via the executable file (not from the shortcut), the window position is restored correctly. How can this be fixed? It is very annoying.

Classic Windows shortcut issue: the first `ShowWindow` can honor the shortcut’s “Run” state instead of the app’s. I'll inspect the restore and elevation paths and fix them.

I'll fix restore so it doesn't honor the shortcut's show state, and stop elevation relaunch from propagating that state.

This is a classic Windows quirk, not a bug in your saved bounds.

**Cause:** On Windows, the **first** `ShowWindow` for a process can ignore the app’s show request and use the launcher’s `STARTUPINFO` instead. A shortcut often passes “Minimized” / “Maximized” / `SW_SHOWDEFAULT`; a direct `.exe` launch usually passes `SW_SHOWNORMAL`. With `StartHidden: true`, that first real show was your restore path — so shortcuts could open minimized and leave placement wrong.

**What changed:**
1. **`restoreWindowOptions`** — after show, force normal (unminimize / unmaximize), then re-apply saved position/size
2. **Elevation relaunch** — use `SW_SHOWNORMAL` instead of `SW_SHOWDEFAULT` so UAC relaunch doesn’t inherit the shortcut’s Run state
3. **Save** — don’t persist geometry while minimized (same as maximized)

Rebuild and launch from the shortcut again; the window should restore consistently.

Optional: right‑click the shortcut → **Properties** → **Run: Normal window** — that helps other apps too, but the app should no longer depend on it.

The minimize fix was the wrong root cause — this looks like a multi-monitor coordinate / restore issue when launched from a shortcut. Digging into how position is saved and applied.

Wails `SetPos` adds the current monitor's origin to coordinates that `GetPos` already returns as absolute — that would pin the window to whichever monitor the shortcut opens on. Checking the earlier Win32 fix and why it was reverted.

Confirmed: Wails `WindowGetPosition` returns absolute screen coords, but `WindowSetPosition` adds the current monitor’s work-area origin. Shortcut launches place the window on the cursor’s monitor (often #3), so restore stays there. Compensating for that asymmetry on Windows only—no HWND enumeration that broke close last time.

The real bug is a Wails Windows API mismatch, not the shortcut “Run” state.

**What happens**
- `WindowGetPosition` saves **absolute** screen coordinates (e.g. `x: 3853`)
- `WindowSetPosition` treats those values as **offsets from the current monitor’s work area** and adds that monitor’s origin

A shortcut usually creates the window on the monitor under the cursor (your third display). Restore then does `monitor3.origin + savedX`, so the window stays on monitor 3. A direct `.exe` launch often starts on the primary monitor (`origin ≈ 0,0`), so the same absolute numbers happen to work.

**Fix:** on Windows, probe the work-area origin, then apply `saved − origin` so the final placement is the absolute saved position—same for shortcut and executable.

Rebuild and test both launch paths. No need to change the shortcut properties.
