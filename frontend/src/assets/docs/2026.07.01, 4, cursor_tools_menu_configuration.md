I verified the legacy `regeditjump` against the current project and found it was **only partially reproduced, with a real bug**.

### What was wrong

Both registry-open paths — `backend/toolsmenu/exec_windows.go` (`platformOpenRegistry`) and the focused `backend/tracemanager/actions_windows.go` (`platformOpenRegedit`) — used a **LastKey-only** shortcut: write `HKCU\...\Applets\Regedit\LastKey`, then launch `regedit.exe`. That reproduces only the *fresh-launch* half of `regeditjump`. The bug:

- If **regedit is already running**, launching `regedit.exe` again just re-activates the existing window and does **not** re-read `LastKey` — so navigation silently does nothing. The user lands wherever regedit already was.
- The legacy `regeditjump`'s actual navigation engine (`regeditapp_t`: find/launch the window, collapse the tree, then expand/type down the path via `WM_KEYDOWN`/`WM_CHAR`, plus value selection in the list view) was never ported.

### What I did

Created a shared, faithful port and wired both callers to it:

- **`backend/winregedit/jump_windows.go`** — `Jump(keyPath)` that:
  - resolves short/long hive prefixes (`HKLM`/`HKEY_LOCAL_MACHINE`, etc.) and normalizes slashes;
  - detects whether the final segment is a key or a **value** (selecting the value in the list view, like the original);
  - writes `LastKey` for the fast fresh-launch case; **and**
  - drives regedit's `SysTreeView32` directly with the same collapse-then-expand/type keystroke sequence as the legacy `navigatetreeview`/`navigatelistview`, so it works when regedit is **already open** (with pacing delays mirroring `REGEDITSLOWWAIT`).
- **`backend/winregedit/jump_other.go`** — non-Windows stub.
- **`toolsmenu/exec_windows.go`** and **`tracemanager/actions_windows.go`** now both call `winregedit.Jump`, removing the duplicated LastKey code.
- Updated `tools/tools.md` and the frontend doc to describe the corrected two-state navigation.
