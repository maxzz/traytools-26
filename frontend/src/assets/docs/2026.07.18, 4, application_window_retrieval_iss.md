## Load path on startup

When `mainTab` is restored as `"windows-tree"`:

1. `Page_WindowsTree` mounts and calls `refreshWindowTree()` if `root === null`
2. Refresh sets `loading = true`, awaits `windowTreeBus.getTree()`, writes `windowTreeStore.root`
3. `useFilter()` (in the toolbar) copies/filters that root into `filteredTreeAtom` via `useEffect`
4. `WindowTreeView` renders from `filteredTreeAtom`

## Likely failure modes (ranked)

### 1. First fetch fails once — no automatic retry (best match for “sometimes at startup”)

`loaded` is `root !== null`. The effect depends only on `loaded`.

- Fetch fails (e.g. `"Backend is not available."` before Wails bindings are ready) → `root` stays `null` → `loaded` stays `false`
- Effect does **not** run again (dependency never changed)
- Spinner stops; UI shows error text or `"No windows. Press Refresh."`
- Manual Refresh works — same call, later, succeeds

That matches intermittent startup timing.

### 2. List is decoupled from fetch success

Even when `windowTreeStore.root` / `count` are set, the view still shows empty until `useFilter`’s effect updates `filteredTreeAtom`. If that sync is delayed or skipped, you get: spinner ran, store has data, list still empty.

Quick check when it happens: open Tree options — if **Total windows > 0** but the pane still says “No windows”, this is the bug.

### 3. Overlapping refreshes with no request id

`refreshWindowTree` has no generation guard. A later failed call does:

```45:47:frontend/src/components/2-main/1-tab-windows-tree/a-windows-tree-calls.ts
        windowTreeStore.error = String(e);
        windowTreeStore.root = null;
        windowTreeStore.count = 0;
```

Dev `StrictMode` double-invokes the mount effect; the View menu also calls `refreshWindowTree()` when switching to the tab. A late failure can wipe a good result.

### 4. Less likely: hung / very slow `getTree`

Backend enumeration is synchronous under the `windowtree` bus mutex. On a busy desktop the spinner can stay up a long time; that looks stuck but is usually just slow, not empty-after-load.

## How to tell which case you hit

Next time it fails, note which of these is true:

1. **Red error** → fetch threw; almost certainly (1) or (3)
2. **“No windows” + Total = 0** → fetch returned empty / was cleared
3. **“No windows” + Total > 0** → (2) filter sync
4. **Spinner never stops** → (4) hung/slow backend call

I have not changed code yet. If you want a fix next, the highest-value changes are: retry/wait for backend on first load, render from `root` (or sync filter during render instead of a later `useEffect`), and add an in-flight request id so stale refreshes cannot clear a good tree.
