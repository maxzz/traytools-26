**User**

After clicking the application icon in the System Tray and selecting the "Show" menu item, subsequent clicks on the icon do nothing, and the window remains on the screen; however, before clicking "Show," the menu appears as expected. Once "Show" has been selected, the menu no longer appears when clicking the icon—this is highly inconsistent behavior that needs to be fixed.

**Cursor**

I'll investigate the system tray show/menu click handling to find why the menu stops appearing after "Show".

Root cause looks solid: Show/toggle run synchronously inside the tray `wndProc`. I'll apply the same async pattern already used for Exit.

Fixed in `backend/tray_windows.go`.

**Cause:** "Show" and left-click toggle called Wails `WindowShow`/`WindowHide` **synchronously** inside the systray `wndProc` (while `TrackPopupMenu` was still unwinding). That stole focus mid-menu and wedged later tray clicks. Exit already avoided this by quitting on a goroutine.

**Fix:** Run `showWindow` / `toggleWindow` with `go`, same pattern as `RequestExit`.

After rebuilding, verify: right-click → Show → right-click again (menu should appear), and left-click toggle should keep working.

The earlier goroutine change wasn’t the real issue. This was the classic Win32 tray-menu bug ([KB 135788](https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-trackpopupmenu)): `energye/systray` called `SetForegroundWindow` + `TrackPopupMenu` but never followed with `PostMessage(WM_NULL)`. After the first open, the next menu flash-dismisses or never shows.

**Fix:** vendored a patched copy at `third_party/energye-systray` and wired it via `go.mod` `replace`. `ShowMenu` now posts `WM_NULL` after `TrackPopupMenu`.

Rebuild and retest: open the tray menu, dismiss it, open it again — it should keep working.
