Implementing Wails `SingleInstanceLock` and an early Windows check before elevation so a second launch doesn't trigger UAC.


Single-instance enforcement is in place using Wails’ built-in `SingleInstanceLock`, plus an early Windows check so a duplicate launch activates the running app instead of triggering UAC elevation again.

### How it works

1. **Early exit (Windows)** — `EnsureSingleInstanceOrExit()` runs before elevation. If another instance is already running, it sends a message to that instance and exits immediately.
2. **Wails single-instance lock** — `SingleInstanceLock` ensures only one process owns the app mutex on Windows, Linux, and macOS.
3. **Window activation** — When a second launch is detected, `OnSecondInstanceLaunch` calls the existing `showWindow()` helper, which unminimizes, shows, and focuses the main window.

### Files changed

- `main.go` — wires up the early check and `SingleInstanceLock`
- `backend/singleinstance.go` — shared config and callback
- `backend/singleinstance_windows.go` — pre-elevation duplicate detection on Windows
- `backend/singleinstance_other.go` — no-op on non-Windows platforms

### Notes

- Single-instance is **disabled for `wails dev`** binaries (same rule as auto-elevation), so dev sessions can still run multiple instances.
- With **Run elevated** enabled, a second shortcut click now activates the existing window instead of showing UAC again.

To verify: build and run the release `.exe`, hide the window to the tray, then launch it again — the existing window should come to the front with no second process.
