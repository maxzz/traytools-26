# Tab selection restoration issue
_Exported on 7/20/2026 at 19:46:29 PDT from Cursor (3.12.17)_

---

**User**

When the "Copy Operations" or "Menu Editor" tab is selected and an application operation require elevation change—causing the application to restart—the currently selected tree item in the active tab must be restored after the restart.

---

**Cursor**

I'll explore how elevation restarts and tree selection work in Copy Operations and Menu Editor so we can restore the selected item after restart.

Next I'll inspect both editors' localStorage and selection logic so we can persist a stable selection path across elevation restarts.

Persisting a stable index-based selection path in each editor's localStorage cache and restoring it after config load (uids are regenerated on file load).

Simplifying Menu Editor init, then verifying the final files.

Tree selection now survives elevation restarts for both editors.

**Why it broke:** Runtime uids (`c1`, `n2`, …) are regenerated when `copy.json` / `tools.json` reload, so a stored uid no longer matches after restart. The active tab already persisted via `appSettings.mainTab`.

**Fix:** Persist a stable index path in each editor’s localStorage cache, then remap it to the new uid after `ensureUids`:

- **Copy Operations** — `{ kind: "root" }` | `{ kind: "group", groupIndex }` | `{ kind: "item", groupIndex, itemIndex }`
- **Menu Editor** — `number[]` (`[]` = root, `null` = none)

Selection is written on every store change and restored on init and after `CopyConfig_Set` / `ToolsConfig_Set` (including post-elevation file load).
