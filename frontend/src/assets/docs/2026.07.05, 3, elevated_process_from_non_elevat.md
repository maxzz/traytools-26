Yes — you can launch an elevated child from a non-elevated process, but **`exec.Command` will not do that for you**. You need an explicit UAC elevation path, and for `winregedit` that has an important follow-on consequence.

## Why `exec.Command` fails

`exec.Command("regedit.exe").Start()` uses `CreateProcess`, which always starts the child at the **same integrity level** as the caller. If the target executable’s manifest says `requireAdministrator`, Windows returns:

**`ERROR_ELEVATION_REQUIRED` (740)** — “The requested operation requires elevation.”

There is no automatic retry; you must handle that yourself.

## How to elevate: `ShellExecute` with `"runas"`

The standard pattern is `ShellExecuteW` with verb `"runas"`, which triggers the UAC prompt. Your project already uses this in `elevation_windows.go`:

```35:66:c:\y\w\2-web\0-dp\utils\traytools-26\backend\elevation_windows.go
	verb, err := windows.UTF16PtrFromString("runas")
	// ...
	ret, _, err := shellExecuteW.Call(
		0,
		uintptr(unsafe.Pointer(verb)),
		uintptr(unsafe.Pointer(exePtr)),
		paramsArg,
		uintptr(unsafe.Pointer(dirPtr)),
		uintptr(windows.SW_SHOWDEFAULT),
	)
```

For regedit, the same idea would be:

```go
ShellExecuteW(0, "runas", "regedit.exe", nil, nil, SW_SHOWNORMAL)
```

Typical return codes:

| Code | Meaning |
|------|---------|
| `> 32` | Success (UAC approved, process started) |
| `1223` (`ERROR_CANCELLED`) | User declined UAC |
| `740` | Elevation still required (unusual with `"runas"`) |

You cannot elevate silently — the user must approve UAC (unless policy auto-approves).

## Detecting elevation required and retrying

You can catch error 740 from `exec.Command` and fall back to `"runas"`:

```go
if err := exec.Command("regedit.exe").Start(); err != nil {
    var errno syscall.Errno
    if errors.As(err, &errno) && errno == windows.ERROR_ELEVATION_REQUIRED {
        return launchRegeditElevated() // ShellExecute "runas"
    }
    return fmt.Errorf("winregedit: failed to launch regedit: %w", err)
}
```

Or call `ShellExecute` with `"open"` first and retry with `"runas"` on failure — `toolsmenu/exec_windows.go` already uses `"open"` for generic launches.

## Critical caveat for `winregedit`

Even if elevated regedit starts successfully, **your keystroke navigation will likely break** when your app is non-elevated and regedit is elevated.

Windows **UIPI** (User Interface Privilege Isolation) blocks a lower-integrity process from:

- `SendMessage` / `PostMessage` to an elevated window  
- `SetForegroundWindow` / `SetFocus` on an elevated window  

So this path in `navigate()` would fail against an elevated regedit:

```201:216:c:\y\w\2-web\0-dp\utils\traytools-26\backend\winregedit\jump_windows.go
	for i := 0; i < 30; i++ {
		sendMessage(tree, wmKeyDown, vkLeft, 0)
	}
	// ...
		sendMessage(tree, wmChar, uintptr(toUpper(regPath[i])), 0)
```

What still works from non-elevated:

- **`LastKey` fast path** — you write `HKCU\...\Regedit\LastKey` before launch; a **fresh** elevated regedit should restore that key on startup.
- **Already-open regedit** — if an elevated instance is already running, you cannot drive it via messages from a non-elevated app.

## Practical options

1. **Keep regedit non-elevated** — on most Windows installs, `regedit.exe` launches fine without admin; only some keys need elevation to edit.
2. **Elevate only when needed** — try `exec.Command` first; on 740, use `"runas"` and rely on `LastKey` (skip keystroke nav for that launch).
3. **Run your app elevated** — you already support `RunElevated` in settings; then `exec.Command` and window messaging both work against elevated regedit.
4. **Split responsibilities** — elevated helper exe that only launches regedit (with `LastKey` pre-written), while the main app stays non-elevated.

**Bottom line:** Yes, `ShellExecute` + `"runas"` is the right way to start an elevated child from a non-elevated parent. For this package, elevation solves launch but conflicts with the existing window-message navigation unless you depend on the `LastKey` path or run elevated yourself.
