`quitOnClose` is on, and `BeforeClose` blocks on `stopTray()` — that matches intermittent slow/stuck closes. Checking whether `systray.Quit` can deadlock the UI thread.

Root cause: `BeforeClose` blocks the UI thread on `stopTray()` (up to 2s). If that times out, `sync.Once` prevents a retry and the systray goroutine can keep the process alive — matching slow, fast, and stuck close.

Also skipping the DevTools window scan during close — it runs on the UI thread and can add unnecessary delay.

The title-bar Close path was blocking the UI thread on tray teardown.

**Cause:** With “Quit on close” enabled, `BeforeClose` called `stopTray()` while handling `WM_CLOSE`. That wait (up to 2s) froze the message pump. If the wait timed out, `sync.Once` prevented another tray quit attempt, and the systray goroutine could keep the process alive — so close felt fast, slow, or stuck.

**Fix:**
1. `BeforeClose` only saves options and allows quit — no tray wait on the UI thread  
2. Tray cleanup runs in `Shutdown` after the window is gone  
3. `os.Exit(0)` after `wails.Run` so a stuck tray goroutine can’t hold the process open  

Rebuild (`p build`) and try the title-bar Close button again — it should dismiss immediately and exit reliably.
